🎥 Video Funnel Media Architecture

Recommended decision: keep one canonical video library in Cloudflare R2 + Cloudflare CDN (or AWS S3 + CloudFront if the client already standardises on AWS), store metadata in a small media registry, and let each channel adapter decide how to deliver the asset. For WhatsApp, send videos as native WhatsApp video messages using either a Meta media_id cache or the CDN link. Do not use Shopify Files as the source of truth for this funnel.

Why this is the right architecture

  • One source of truth. The same clip can be used by WhatsApp, the existing AI web funnel, landing pages, email, and future channels without uploading separate permanent copies everywhere.
  • WhatsApp-native playback. The bot can send a type: "video" message. The user sees a preview in WhatsApp and watches inside the chat, instead of opening a web page.
  • Future-proof. Meta media_id values are treated as a temporary delivery cache, not the permanent library. If an ID expires or fails, the backend re-uploads from R2/S3 or sends by CDN link.
  • Cheap at scale. For many repeated 15–30 second clips, R2 avoids bandwidth surprises and keeps the system simpler than per-channel asset storage.
  • Operational control. We can version files, validate encoding, track which funnel step uses which clip, replace a clip without changing the bot logic, and run health checks before campaigns.

Storage Options Compared

OptionUse CaseProsConsRecommendation
Shopify Files Simple product-page assets inside Shopify. Easy for store admins; Shopify CDN works for storefront media. Weak as an omnichannel media backend: limited workflow/versioning, less control over signed access, transcoding, validation, and API-driven WhatsApp reuse. Do not use as source of truth. OK only for store-page copies if needed.
AWS S3 + CloudFront Enterprise-grade canonical storage. Very stable, mature permissions, lifecycle rules, great tooling. More setup; egress/CDN cost must be watched; heavier than needed for 20–50 short clips. Good if the client already uses AWS.
Cloudflare R2 + CDN Canonical video library for WhatsApp + AI funnel. Low storage cost, no standard egress fees from R2, easy CDN, simple public/signed delivery. Needs a small admin/upload workflow and metadata registry. Best default for this project.
WhatsApp Media ID only Temporary channel cache for delivery. Fast to send after upload; avoids making Meta fetch the public URL every time. Not permanent storage. Uploaded media IDs expire; cannot be the single library for web/email/AI channels. Use as cache, not source of truth.

Recommended Media Pipeline

  1. Upload canonical video to R2/S3 under a stable path like whatsapp/dans-backpack/demo-spraying-v1.mp4.
  2. Validate and encode before publishing: MP4, H.264 Main/Baseline, AAC audio, single audio stream, -movflags faststart, target under 16 MB.
  3. Create a media registry row with fields such as asset_key, title, funnel_step, cdn_url, duration, size_bytes, sha256, whatsapp_media_id, media_id_expires_at, and status.
  4. When the funnel chooses a video, the WhatsApp adapter checks whether a valid whatsapp_media_id exists. If yes, send by id. If not, upload the canonical R2/S3 file to WhatsApp, save the new ID, then send.
  5. Fallback path: if upload/cache fails, send the WhatsApp video message by HTTPS link from the CDN, so the user still receives an in-chat video message.
  6. Reuse the same registry for the existing AI funnel: the web/chat channel uses cdn_url, while WhatsApp uses media_id or link.

WhatsApp Video Rules to Design Around

  • Native attachment is supported. The Cloud API media object supports id or link for video. This is not just a plain URL message.
  • Size limit: video messages should be kept under 16 MB.
  • Formats: MP4 (video/mp4) or 3GPP; for MP4 use H.264 video + AAC audio.
  • Compatibility: avoid H.264 High profile with B-frames because it can fail on Android WhatsApp clients. Encode Main/Baseline and use faststart.
  • Media IDs are temporary. Uploaded WhatsApp media persists for about 30 days, so we cache and refresh IDs instead of relying on them as permanent storage.
  • Templates: if the bot initiates a message outside the 24-hour service window, the outbound message must be an approved template. Media can be used in approved template headers, but template copy/media rules should be tested during setup.
  • Interactive flow limits: simple reply buttons are limited, so plan concise choices. For many branches, use list messages or stateful text commands rather than sending too many buttons.

Client-Facing Answer in Russian

Короткий ответ: лучше не выбирать между Shopify Files и WhatsApp Media ID как постоянным хранилищем. Правильная архитектура — отдельная медиатека в Cloudflare R2 или AWS S3, а WhatsApp Media ID использовать только как кэш доставки.

Рекомендуемая схема:

  1. Все оригинальные ролики DAN’S Airless Backpack хранятся в одном месте: Cloudflare R2 + CDN. Альтернатива — AWS S3 + CloudFront, если вся инфраструктура уже в AWS.
  2. В базе/таблице хранится реестр видео: ключ ролика, название, шаг воронки, CDN-ссылка, размер, длительность, версия, WhatsApp media_id и срок его действия.
  3. WhatsApp-бот отправляет ролик как полноценное видео-вложение через WhatsApp Cloud API, а не как текстовую ссылку. Пользователь видит превью и смотрит видео прямо в чате.
  4. Если для ролика уже есть актуальный WhatsApp media_id, бот отправляет его по ID. Если ID истек, backend автоматически загружает ролик заново из R2/S3 в WhatsApp и сохраняет новый ID.
  5. Та же медиатека используется в существующей AI-воронке: WhatsApp берет media_id/link, веб-воронка берет CDN URL. Дублировать ролики в нескольких системах не нужно.

Ответы на вопросы клиента:

  • Где хранить видео? Cloudflare R2 + CDN как лучший вариант по цене и простоте. AWS S3 + CloudFront — хороший вариант, если уже используется AWS. Shopify Files — не лучший источник правды для омниканальной AI-воронки.
  • Можно ли отправлять не ссылкой, а вложением? Да. WhatsApp Cloud API умеет отправлять видео как native video message через video.id или video.link. В чате будет превью, ролик будет проигрываться внутри WhatsApp.
  • Нужно ли загружать напрямую в WhatsApp и использовать Media ID? Да, но как кэш. Media ID удобен для часто используемых роликов, но не является постоянным хранилищем. Основной файл должен оставаться в R2/S3.
  • Самый быстрый, стабильный и дешевый вариант? R2/S3 как постоянное хранилище + автоматический кэш WhatsApp Media ID + fallback на CDN link. Это дает контроль, масштабирование и отсутствие зависимости от одного канала.
  • 20–50 видео — нормально? Да, это небольшой объем. Главное — держать каждый WhatsApp-ролик до 16 MB, правильно кодировать MP4 и вести реестр версий.

Итоговая рекомендация: строим не “WhatsApp-хранилище”, а общую video asset layer для всех каналов. WhatsApp получает оптимизированные ролики из этой библиотеки, AI-воронка использует те же asset keys, а замена или обновление видео делается один раз в центральной медиатеке.

⚠️ Edge Cases, Bottlenecks & Complications

  1. The 24-hour customer service window. Once a customer messages you, you can reply freely for 24h. After that, every outbound message MUST be a pre-approved template. Forgetting this kills automation flows.
  2. Template approval delays. Each template needs Meta approval, usually 24–72 hours. New product launches / sales need templates submitted well in advance. Rejection reasons can be vague ("too promotional"). Plan a template library.
  3. Template categories & pricing. Marketing > Utility > Authentication in cost. Mis-categorising leads to rejection or higher bills. Australia is a Tier-1 market = highest pricing tier.
  4. Quality rating. Meta tracks user complaints. Too many "report spam" / blocks drops your number to Medium → Low → Flagged → permanent ban. Always offer easy opt-out.
  5. Number portability is painful. Choose the WhatsApp number carefully — switching providers later (Twilio ↔ Meta direct) is multi-day downtime.
  6. One number = one WABA. Cannot share a number across Twilio + Meta direct. If we ever want to migrate, must "release" then "re-register".
  7. Voice notes from customers. Customers love sending voice notes. Need STT pipeline (ElevenLabs scribe or OpenAI Whisper) to feed text into the agent.
  8. Image messages. Customers send paint colour photos or pictures of damaged walls. Need a vision model (GPT-4o, Claude) to interpret. Adds cost per image.
  9. Latency. WhatsApp users expect responses within seconds. Each message hop is: Meta → Vercel webhook → ElevenLabs API → Vercel → Meta. Cold starts on Vercel can push this to 5–10s. May need warm functions or edge runtime.
  10. Concurrent conversations. 50 customers messaging at once. Need a session store keyed by their phone number, not a single global state.
  11. Out-of-hours scenarios. When a human handoff is needed but the team is asleep. Define an explicit "we're closed, leave a message" template + Shopify flag.
  12. Misdirected messages. People send "hi" / "test" / wrong number. Need an idle/noise filter so you don't burn template credits replying to noise.
  13. Phone-only customers. Many WhatsApp leads will never give email. Shopify customer record will be phone-only — but Shopify customers historically need email or phone (now either works). Decide what's the minimum to count as a "lead".
  14. Meta business verification rejections. Common reasons: business address mismatch, ABN not visible on documents, website not linked. Plan time for re-submissions.
  15. Opt-in compliance. Australian Spam Act + Meta policy: outbound to a number requires explicit opt-in (e.g., checkbox at checkout, or the customer messaged us first). Cannot import contact lists and blast.
  16. Meta API versioning. Cloud API breaks at version bumps (~yearly). Need to monitor deprecation notices.
  17. Pixel + CAPI dual-tracking deduplication. When firing events both client-side (Pixel) and server-side (CAPI), missing the event_id on both sides causes double-counting in Ads Manager.
  18. iOS/Android quirks. Interactive buttons render differently. Test on both. Long messages get clipped on older Androids.
  19. Catalogue sync (if used). Meta Catalogue must be kept in sync with Shopify products. Either via Meta's Shopify integration app or a custom feed. Stale prices = customer complaints.
  20. Cost spiral. Each conversation triggers: WhatsApp API charge + ElevenLabs text API + Shopify API + (optional) vision/STT. Easy to hit $0.20–$0.50 per conversation if not careful.
  21. Video size failures. WhatsApp video delivery fails when files exceed supported limits or use unsupported encoding. Every funnel asset must pass a preflight check before it is mapped to a bot step.
  22. Expired WhatsApp media IDs. Media IDs should be cached with expiry and refreshed from the canonical R2/S3 file. Never hard-code a media ID as the only copy of a video.
  23. CDN access for Meta. If sending by link, Meta must be able to fetch the HTTPS URL. Private buckets need signed URLs that last long enough for Meta to retrieve the file.
  24. Privacy / GDPR / Australian Privacy Act. Storing transcripts of WhatsApp conversations needs a clear privacy policy update.
