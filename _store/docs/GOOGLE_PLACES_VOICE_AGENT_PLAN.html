Google Places + AI Voice Agent - Full Implementation Plan

Paint Access AI Agent - Generated 25 May 2026

What You're Asking For

During a live phone or website voice conversation, the AI agent should understand a request like:

Customer: "I need a painter near William Street Sydney."

The system should search Google Maps / Google Places in real time, find relevant nearby businesses, rank them, and return the best option to the agent so it can say something useful out loud.

  1. Extract intent - trade/service type, address/suburb, optional radius, urgency.
  2. Call Google Places - search for matching businesses near the address.
  3. Score results - combine distance, rating, review count, business status, and optional review recency signal.
  4. Return a short spoken answer - name, rating, review count, address, phone/website if available.
  5. Avoid overclaiming - Google cannot tell us which individual worker did a job last week; that needs CRM/job data later.
Important expectation: Google Places is a business-directory approximation. It can help choose "best nearby business" using public rating/review signals. It cannot guarantee availability, quality for a specific job, whether the business services that exact suburb, or which worker performed a recent job.

Google Places API Approach

Option A - Text Search First (Recommended MVP)

How it works: Send the whole natural query to Places Text Search, e.g. painters near William Street Sydney NSW.
  • Fastest to build.
  • Handles spoken, messy, human-style queries well.
  • Good for service categories that do not map cleanly to one place type.
  • Less precise control over exact radius unless we add location bias/restriction.

Option B - Geocode Then Nearby Search

How it works: Convert address to lat/lng, then run Nearby Search around that coordinate.
  • Better distance control.
  • Useful when customer gives a specific street/suburb/postcode.
  • More moving parts: Geocoding + Places search + optional Details calls.
  • Trade categories may not always map perfectly to Google included types.

Recommendation

Start with Text Search for the MVP, because the voice agent receives natural language. Add Geocoding/Nearby Search only if the Text Search ranking is not good enough in testing.

Ranking Logic

The AI should not simply read the first Google result. We should rank candidates with a deterministic score and give the agent a short explanation.

score =
  rating_score        (0-40 points)
+ review_count_score  (0-25 points)
+ distance_score      (0-20 points)
+ activity_score      (0-10 points, optional)
+ business_status     (0-5 points)
- penalties           (closed, too few reviews, weak match, too far)
SignalRule
RatingPrefer 4.5+; reject or de-prioritise below 4.0 unless there are no alternatives.
Review count20 reviews minimum for confidence; 100+ reviews is stronger.
DistancePrefer close to requested address, but do not choose a weak 5-review business over a strong nearby business.
ActivityIf reviews are requested and timestamps exist, boost places with recent review activity. Do not over-weight this because Google exposes limited review data.
Open/closed statusPrefer operational businesses. If opening hours are available, mention if it may be closed now.
Category matchPrefer businesses whose name/types match the requested service: painter, painting contractor, decorator, etc.

Phased Implementation Plan

Phase 0 - Client/API Setup
  • Create or confirm a Google Cloud project.
  • Enable Google Maps Platform billing.
  • Enable Places API (New), and optionally Geocoding API.
  • Create a restricted server-side API key.
  • Add GOOGLE_MAPS_API_KEY to Vercel environment variables.
  • Set a daily/monthly budget alert in Google Cloud.
Phase 1 - Backend Endpoint
  • Add api/google/places-specialist.js.
  • Reuse existing bearer auth via API_SECRET_TOKEN.
  • Validate input: service, location, radius.
  • Call Google Places Text Search with a minimal field mask.
  • Shape response into compact JSON for the voice agent.
  • Add clear fallbacks when location or service is missing.
Phase 2 - Ranking + Contact Details
  • Score top candidates by rating, review count, distance, and business status.
  • Optionally call Place Details for the top 3 only when phone/website is requested.
  • Add result caching to reduce duplicate calls during repeated testing.
  • Return best result plus 2 alternatives for the agent to use if needed.
Phase 3 - ElevenLabs Tool + Prompt Rules
  • Create a new webhook tool: find_local_specialist.
  • Attach it to the current ElevenLabs agent.
  • Update the system prompt with rules: when to call the tool, how to ask for suburb/address, and how to avoid overclaiming.
  • Use pre-tool speech so the caller hears a natural filler while the API call runs.
Phase 4 - Testing
  • Test 20-30 realistic queries: painters, plasterers, cleaners, roofers, electricians, handymen.
  • Test vague locations: "near me", "Sydney CBD", "William Street", "Bondi", "Melbourne".
  • Test bad inputs: no trade, no location, impossible address, overseas address.
  • Run live voice tests for latency and phrasing.
  • Confirm the result does not sound like a guaranteed endorsement.
Phase 5 - Production Hardening
  • Add structured logs without leaking API keys or unnecessary personal data.
  • Add rate limits and timeout handling.
  • Add cache TTL and query normalisation.
  • Add budget protection: fail gracefully if Google quota/billing is unavailable.
  • Optional: add dashboard/report of recommendations made by the agent.

Bottlenecks, Edge Cases & Complications

  1. Google API billing and quotas. No API key/billing means no real implementation. Budget alerts are mandatory before live use.
  2. Field-mask pricing surprises. Rating, review count, phone, website, and reviews can cost more than basic place IDs/address fields. We must keep response fields tight.
  3. Voice latency. Live calls cannot tolerate 8-15 second silence. The endpoint should return in 2-4 seconds where possible, and the agent must speak a filler before tool execution.
  4. Ambiguous spoken addresses. "William Street Sydney" could mean a street, a business, or a region. The agent may need to ask for suburb/postcode if confidence is low.
  5. Trade taxonomy mismatch. Customer says "painter"; Google categories may include painting contractor, decorator, home goods, contractor, etc. Query expansion is needed.
  6. Review recency is not a full signal. Google may expose only limited review samples. We can approximate activity but cannot audit full freshness.
  7. Sponsored/SEO-biased results. Google ranking is not the same as Paint Access quality. Our ranking should be transparent and conservative.
  8. Fake or inflated reviews. High rating/review count is useful, but not a guarantee. The agent should say "Google lists..." rather than "the best painter is...".
  9. Availability not known. Google Places does not tell us whether the business can take a job today.
  10. Business service area mismatch. A business may appear nearby but not service the customer's exact address.
  11. Closed or moved businesses. Google data can lag. Filter by business status where available.
  12. Privacy and compliance. If customer address is sent to Google, privacy policy may need to mention third-party API processing.
  13. API key security. The key must stay server-side in Vercel env vars, restricted by API, and never exposed in storefront JavaScript.
  14. Too much information aloud. The agent should read one best result and ask whether the customer wants phone/website, not list five businesses.
  15. Cost creep during testing. Repeated test calls can generate Places charges. Use caching and test fixtures where possible.
  16. Legal/reputational risk. Paint Access may not want to recommend third-party trades without a disclaimer. Client must approve wording.

Default search radius - recommended starting point: 5 km in metro areas, 15-25 km in regional areas.
Recommendation wording approval - especially disclaimers like "Google lists this business highly; please confirm availability directly."
Decision on phone/website - should the agent always fetch contact details, or only after customer asks? Fetching only on request is cheaper.
Privacy policy approval - confirm whether sending customer-provided addresses to Google is acceptable under current policy.
Test call scenarios - 10-20 real examples the client expects customers to say.

Questions To Clarify Before Building

Q1. Business goal. Is Paint Access recommending independent trades to help customers, or should the agent only provide neutral "nearby options from Google" without endorsement?
Q2. Category list. Which specialist types are in scope for v1? Painter only, or multiple trades?
Q3. Geography. Should searches be limited to Australia, NSW, or Sydney?
Q4. Ranking preference. Should the system prefer highest rating, most reviews, closest distance, or a balanced score?
Q5. Minimum quality threshold. Should we hide businesses below 4.0 rating or below 20 reviews?
Q6. Contact details. Should the agent offer phone/website immediately, or ask first?
Q7. Review freshness. Is review recency important enough to pay for review fields, or is rating + review count enough for MVP?
Q8. CRM future. Will Paint Access later provide a list of preferred contractors/jobs so we can rank internal trusted partners above public Google results?
Q9. Logging. Should every recommendation be logged for staff review?
Q10. Escalation. If no good result is found, should the agent collect the customer's number for a human follow-up?
